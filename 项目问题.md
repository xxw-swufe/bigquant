## 一、已修复（需重启 Kernel + 重跑 Notebook 后验证）

### 1. 首轮「研究20日动量」误识别 20 个因子

**现象**

- 用户问 `研究20日动量`，返回 `recognized_not_implemented`
- 「已识别因子」列出约 20 个（momentum_5d、momentum_20d、volatility_20d…）
- 用户问题被展示成因子描述长句，而非原始输入

**根因**

- 对话计划被序列化成自然语言描述后，再次送入 `resolve_factor_intent` 做语义扩展
- 长描述命中大量「动量 / 20 / ETF」相关别名，因子被过度展开

**修复**

- 管线改为直接传结构化 `planned_factor_names`，不再二次解析重写文本
- `build_effective_user_idea` 使用因子 canonical name，不用 `description`
- 失败报告只列实际选中因子

**相关文件**：`chat_agent.py`、`agent_tools.py`、`plan_mutation.py`、`factor_resolver.py`

---

### 2. 「改成未来 N 日」计划改了，回测仍用 5 日

**现象**

- 多轮对话中回测数字始终相同（总收益 -4.92%、年化 -10.73%、最大回撤 -35.68%…）
- LLM 把「未来 10 日」理解成「能否预测未来」，或把「未来 5 日」理解成「持有期 2 天」

**根因**

- `agent_tools.tool_run_factor_research_pipeline` 写死 `target_col="future_return_5d"`

**修复**

- 根据 `selection.target.horizon` 动态选择 `future_return_{horizon}d`
- LLM 摘要增加 `target_horizon`、`target_metric`、`selection_status` 字段

**相关文件**：`agent_tools.py`、`chat_agent.py`、`research_plan.py`（`target_return_column`）

---

### 3. 「再加 BBI」未阻断，静默跳过继续回测

**现象**

- 预期：`recognized_not_implemented`，不跑回测，旧计划保留
- 实际：回测照常跑，LLM 说「未包含 BBI 因子」

**根因**

- BBI 进入 `recognized_not_implemented_terms`，但未进入 `factor_names`
- draft 计划不变，管线仍用旧因子执行
- 缺少 chat 层前置拦截

**修复**

- `chat_agent._recognized_not_implemented_block`：本轮识别到未实现因子时直接返回失败报告

**相关文件**：`chat_agent.py`、`chat_state.py`

---

### 4. 「再加相对强度」报 `missing_fields` 而非缺 benchmark

**现象**

- 预期：`missing_context`，提示选择沪深300 / 中证全指等 benchmark
- 实际：`data_unavailable` + `relative_strength_20d: missing_fields`

**根因**

- 无 benchmark 时 `data_features` 仍会生成全 NA 的 `relative_strength_20d` 列
- 可用性检查把 `benchmark_close` 等当成缺列，未区分「缺 benchmark 上下文」

**修复**

- 区分 `available_fields`（数据列）与 `available_context`（如 `benchmark_series`）
- benchmark 依赖因子优先报 `missing_context`
- `_missing_context_reply` 引导用户选择 benchmark

**相关文件**：`factor_generator.py`、`factor_availability.py`、`agent_tools.py`

---

### 5. Notebook 每条消息 / 回答出现两遍

**现象**

```
你：研究20日动量和低波动
你：研究20日动量和低波动
回答：……
回答：……
```

**根因（双重）**

1. **旧按钮监听未注销**  
   `importlib.reload(notebook_chat_ui)` 只重置模块内 `_ACTIVE_CHAT_UI`，页面上旧 widget 的 `on_click` 仍在。点一次发送 → 两个 handler 各执行一次。

2. **只 reload UI，未 reload 业务代码**  
   `chat_agent`、`factor_generator` 等仍在 `sys.modules` 旧版本，导致上述 BBI / 相对强度修复在 Notebook 里不生效。

**修复**

- UI 单例改存 IPython `user_ns`，reload 后仍能关闭旧面板
- 旧 handler 检查「是否为当前活跃 UI」，否则直接 return
- 新增 `src/notebook_reload.py`：`reload_autoetf_runtime()` 一次 reload 全部运行时模块
- Notebook 启动格改为调用 `reload_autoetf_runtime()`，不再只 `reload(notebook_chat_ui)`

**相关文件**：`notebook_chat_ui.py`、`notebook_reload.py`、`notebooks/AutoETF_Local_Research_Agent.ipynb`

### 6. VS Code / Cursor 看不到虚拟环境

**现象**

- 解释器列表里没有 `.venv-vscode`
- 项目里「找不到」虚拟环境

**根因**

- `notebooks/yangpeng.code-workspace` 把工作区根目录指到 `/Users/yangpeng`（用户主目录），不是 `ai量化` 项目
- `${workspaceFolder}/.venv-vscode` 解析到错误路径
- `.venv-vscode` 为隐藏目录，默认不易看见

**修复**

- 新建 `AutoETF.code-workspace`，根目录指向项目本身
- 修正 `yangpeng.code-workspace`
- `.vscode/settings.json`、VS Code / Cursor 用户设置写入绝对路径
- 重新注册 Jupyter kernel：`Python (.venv-vscode AutoETF)`

**正确打开方式**：`文件 → 打开工作区 → AutoETF.code-workspace`

## 四、复测检查清单

每次改代码后在 Notebook 中：

1. **Kernel → Restart**
2. 从头运行所有单元格（含 `reload_autoetf_runtime()` 启动格）
3. 确认页面上 **只有一个** 聊天面板
4. 发一条消息，确认 **只有一条「你：」和一条「回答：」**
5. 跑 `量化测试.md` 两套脚本，对照下表

| 检查点 | 预期 |
|--------|------|
| 失败时旧计划 | `committed_plan` 不变 |
| 改周期 | 因子不变，仅 `target.horizon` 变；回测列跟随 |
| 空意图 | 不启动默认 MVP 因子 |
| BBI | `recognized_not_implemented`，不跑回测 |
| 相对强度（无 benchmark） | `missing_context` + benchmark 提示 |
| 删不存在因子 | 「当前计划中没有要删除的因子」 |
| 删到空计划 | 「删除后计划为空，请先明确新的研究因子」 |

# AutoETF Research Agent 方案说明

## 1. 项目定位

AutoETF Research Agent 是一个面向行业 ETF / 主题 ETF 轮动研究的量化研究辅助系统。第一版 MVP 先聚焦条件事件研究：检验 ETF 在单日满足“量比 > 1、换手率 < 5、今日上涨”后，下一交易日上涨概率是否高于全样本基准概率。

项目代码结构会保留多因子研究扩展入口。后续可在同一框架下升级为 ETF 因子生成、单因子有效性检验、因子标准化、多因子综合评分、Top 10% ETF 简化回测、权重合理性诊断和 Markdown 研究报告生成。

本项目仅用于历史回测、统计检验和策略研究，不构成任何投资建议。

## 2. 核心流程

```text
用户自然语言问题
→ 条件规则结构化
→ BigQuant ETF 数据获取
→ 计算量比、换手率、当日收益、次日收益标签
→ 筛选满足条件的 ETF-日期事件样本
→ 统计次日上涨概率与全样本基准概率
→ 分年份稳定性检验
→ 条件有效性诊断
→ 生成 Markdown 研究报告
```

## 3. MVP 范围

第一阶段只做 ETF 日频条件事件研究：

- 条件解析：量比 > 1、换手率 < 5、今日上涨
- 标签定义：下一交易日收益率是否为正
- BigQuant ETF 日线数据获取
- 条件事件样本筛选
- 条件样本上涨概率
- 全样本基准上涨概率
- 概率提升与平均收益提升
- 分年份稳定性
- 条件有效性诊断与研究报告

第一阶段不做新闻、研报、分钟线、深度学习、实盘交易、复杂机器学习和多因子权重优化。

## 4. BigQuant 数据要求

BigQuant 相关逻辑集中放在：

- `src/data_probe_bigquant.py`
- `src/data_loader_bigquant.py`

正式取数前必须先运行字段探测：

```python
from src.data_probe_bigquant import run_data_probe

probe_results = run_data_probe()
```

如果某张表不存在或无权限，程序应降级使用可用表，不应直接崩溃。

## 5. 未来函数约束

特征因子只能使用当前日和历史数据，禁止在特征中使用 `m_lead` 或 `future_return_*`。

`future_return_5d`、`future_return_20d` 只能作为评价标签，不能用于排序、评分或选 ETF。

## 6. 回测说明

第一阶段使用 pandas 简化回测：

- 每个调仓日计算综合得分
- 选择 Top 10% ETF
- 等权持有
- 扣除简化交易成本
- 统计累计收益、年化收益、最大回撤、夏普比率、胜率、换手率

该回测不包含真实撮合、滑点、折溢价、流动性冲击等交易细节。

## 7. 智能体搭建状态与缺口

当前两个 Notebook 已经在 BigQuant AI Studio 中跑通，说明研究底座、平台数据访问和本地开发到平台执行的闭环已经成立。后续可以进入完整智能体搭建阶段，但当前系统仍主要是“研究流程原型”，还不是完整 Agent。

### 7.1 已具备模块

- 问题解析层：`src/condition_parser.py`、`src/agent_parser.py`
- 数据探测层：`src/data_probe_bigquant.py`
- 数据加载层：`src/data_loader_bigquant.py`
- 条件事件研究层：`src/condition_analysis.py`、`src/condition_report.py`
- 因子研究层：`src/factor_generator.py`、`src/factor_analysis.py`、`src/etf_factor_library.py`
- 组合与打分层：`src/scoring.py`
- 简化回测层：`src/backtest.py`
- 诊断与报告层：`src/diagnosis.py`、`src/report.py`、`src/plotting.py`
- 配置层：`src/config.py`
- 运行入口：`notebooks/AutoETF_Condition_Research_MVP.ipynb`、`notebooks/AutoETF_Research_Agent_MVP.ipynb`、`run_condition_mvp_demo.py`

### 7.2 缺少模块

- 统一调度层：缺少类似 `run_autoetf_agent(user_query)` 的总入口，用于接收用户自然语言并自动执行完整研究流程。
- 任务路由层：缺少对用户问题类型的自动判断，例如条件事件研究、多因子研究、数据探测、回测诊断等。
- 工作流编排层：当前流程主要写在 Notebook 中，缺少可复用的 pipeline 函数，例如 `run_condition_research_pipeline()` 和 `run_factor_research_pipeline()`。
- 状态管理层：缺少对表结构探测结果、上次运行配置、用户问题、中间结果和报告路径的统一记录。
- 错误恢复与降级层：缺少对缺失字段、表权限不足、函数名不兼容、基准数据不可用等情况的自动降级策略。
- 统一输出层：缺少标准化结果对象，例如 `AgentResult` 或 `ResearchResult`，用于统一承载假设、数据概况、分析结果、诊断和报告。
- 交互层：缺少 CLI、Web UI 或对话入口。当前主要依赖 Notebook 手动运行。

### 7.3 下一步优先级

下一步最划算的是先做“统一调度层 + 工作流编排层”。

优先实现：

- `run_condition_research_pipeline(user_query, config)`
- `run_factor_research_pipeline(user_query, config)`
- `run_autoetf_agent(user_query, mode="auto")`

这样可以先把已经跑通的 Notebook 流程沉淀为可调用函数，再由统一入口自动选择流程。完成这一层后，再补任务路由、统一结果对象、错误恢复和交互界面，整体返工成本更低。
